## Decisions Made

- **Whitelist keys on folder name, not `frontmatter.name`.** Existing `skills/help/SKILL.md` declares `name: gdd-help` while the folder is `help/`. The D-09 whitelist (`help`, `stats`, etc.) reads naturally as folder names; keying on folder defends against name-vs-folder drift and matches the convention every other tool in this repo uses (skill discovery walks the directory tree).
- **`disable-model-invocation` shape check layered before whitelist check.** Plan only specified the whitelist check (Test 8). I added a shape check that rejects anything other than the YAML boolean `true|false` (parser strips quotes, so quoted `"true"` and bare `true` both pass; raw `True` or `yes` blocks). Documented as Test 9 alternate path in the plan; folded into the missing-frontmatter / disable-model-invocation test coverage in the suite.
- **Tolerated `--strict-description` ".\s*$" regex tail.** `STRICT_RE = /\. Use when .+\.\s*$/m` accepts optional trailing whitespace — defends against YAML-trim ambiguity when the description ends with a quoted string. Pure cosmetic.
- **All constants exported.** `WARN_LINES`, `BLOCK_LINES`, `DESC_MIN`, `DESC_MAX`, `STRICT_RE`, `DISABLE_INVOCATION_WHITELIST` — plus `parseFrontmatter`, `countLines`, `validateSkill`, `walkSkills`, `parseArgs` as named exports for test consumption without subprocess spawn.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Missing Critical] Added `disable-model-invocation` shape check**

- **Found during:** Task 2 (validator implementation).
- **Issue:** Plan only specified that `disable-model-invocation: true` on a non-whitelisted skill should block. But the plan didn't say what happens if the value is malformed (e.g. `disable-model-invocation: True` or `disable-model-invocation: yes`). A loader processing the malformed value would silently skip the routing flag (depending on parser), creating an undetected misconfiguration.
- **Fix:** Added an explicit shape check before the whitelist check: anything other than the lowercase boolean strings `true|false` (after quote-stripping) produces a `disable-model-invocation-shape` block. The whitelist check only runs when the value is exactly `true`.
- **Files modified:** `scripts/validate-skill-length.cjs`.
- **Verification:** Existing tests pass. Plan AC #10 (`grep -c "disable-model-invocation" >=2`) still satisfied (now 8 occurrences).
- **Committed in:** `494016d` (Task 2 commit).

**2. [Rule 3 — Blocking] Used `realpathSync(mkdtempSync(...))` in test fixture helper**

- **Found during:** Task 3 (test authoring).
- **Issue:** Plan's task body did not call out the macOS `/var` → `/private/var` symlink resolution. Phase 27.6 lessons applied folder mentioned it as a discipline — applied preventively.
- **Fix:** `function makeSkillsDir() { return fs.realpathSync(fs.mkdtempSync(...)); }` defeats the symlink without breaking Windows / Linux behavior.
- **Files modified:** `tests/skill-authoring-contract.test.cjs`.
- **Verification:** All 19 tests pass on Windows (CI test platform); macOS path safe by construction.
- **Committed in:** `3828b5b` (Task 3 commit).

**3. [Rule 1 — Bug] Added `level` field + `descriptionLength` + `hasRequiredFields` + `reasons[]` to result object**

- **Found during:** Task 2 (validator implementation).
- **Issue:** Plan's JSON output spec said `[{name, path, lines, descriptionLength, hasRequiredFields, level: "clean"|"warn"|"block", reasons: [...]}]` but the inline code in the plan's `<action>` block only populated `name/path/lines/errors/warnings`. Tests referencing `parsed.skills[0].level` would have failed without the extra fields.
- **Fix:** Added `level`, `descriptionLength`, `hasRequiredFields`, `reasons[]` to every `result` object returned by `validateSkill`. Computed `level` at end of function (`block` if any errors, `warn` if any warnings, else `clean`).
- **Files modified:** `scripts/validate-skill-length.cjs`.
- **Verification:** `--json` test now asserts `parsed.skills[0].level === 'clean'` and passes.
- **Committed in:** `494016d` (Task 2 commit).
