- Intro paragraph framing the vocabulary as a shared-language anti-relitigation device with GDD analogs surfaced inline
- 8 term `##` sections in canonical Ousterhout order:
  1. **Module** — a unit hiding implementation behind an interface; cross-link to `./design-system-guidance.md` for the design-system-level analog
  2. **Interface** — what callers depend on; cross-link to `./component-authoring.md` 6-principle quality standard
  3. **Implementation** — what callers must NOT depend on; with the leak-detection pattern (de-facto interface)
  4. **Depth** — interface-cost vs implementation-power ratio; deep vs shallow worked examples; cross-link to `./component-authoring.md` "Minimal API"
  5. **Seam** — hypothetical (1 impl) vs real (≥2 impls); the boundary where substitution becomes possible
  6. **Adapter** — interface transformer; design-token theme cited as a GDD-flavored example; Phase 27 acp/asp two-adapter realism noted
  7. **Leverage** — work-done vs API-touched ratio; high-leverage primitive multiplier effect
  8. **Locality** — spatial cohesion of change footprint; broken-vs-healthy chart-type worked example tied back to deletion test
- `## Principles` section with all 3 principles named explicitly:
  - **Deletion test** — interface stands without implementation
  - **Interface is the test surface** — implementation churn does not churn tests
  - **One adapter = hypothetical seam; two adapters = real seam** — YAGNI for boundaries
- `## How this applies to skill authoring` sub-section (~12 lines) — connects the vocabulary to the 100-line cap and extract-then-link discipline (D-10) defined in `./skill-authoring-contract.md`
- `## Cross-references` section closes the file with outbound links to: design-system-guidance, component-authoring, skill-authoring-contract, context-md-format, adr-format
