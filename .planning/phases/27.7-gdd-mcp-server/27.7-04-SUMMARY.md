# Phase 27.7 Plan 04: Installer `--register-mcp` extension + `gdd-health` SKILL MCP-status step + dismissable nudge — Summary

Opt-in `gdd-mcp` registration with Claude Code + Codex via `scripts/install.cjs --register-mcp` (idempotent, graceful absent-CLI fallback), plus a `check-mcp-registration` step block in `skills/health/SKILL.md` that surfaces registration status with a `.design/config.json#mcp_nudge=false` dismissal hook.

## What shipped

### `scripts/lib/install/mcp-register.cjs` (new)

Pure CommonJS library. No side effects on `require`. Exports:

| Symbol | Purpose |
| ------ | ------- |
| `registerMcp({harness, mode?, dryRun?, spawnFn?})` | Register gdd-mcp with one harness; returns `{detected, applied, idempotent_skip, command, exit_code, notice?, dry_run?, stdout?, stderr?}` |
| `detectMcpRegistration({spawnFn?})` | Read-only detection across all harnesses; returns `{harnesses: [...], summary}` |
| `detectHarnessPresent(harness, spawnFn?)` | Boolean — runs `<binary> --version` |
| `isAlreadyRegistered(harness, spawnFn?)` | Boolean — parses `<binary> mcp list` for `\bgdd-mcp\b` |
| `buildHarnessCommand(harness, mode?)` | Returns `{binary, args}` for the add or list command |
| `HARNESSES` | `Object.freeze`d record of `claude` + `codex` command shapes |
| `MCP_NAME` | `'gdd-mcp'` |

The harness command shape is hardcoded:

```
claude mcp add gdd-mcp -s user -- gdd-mcp
codex  mcp add gdd-mcp -- gdd-mcp
```

The `--` separator before `gdd-mcp` prevents flag injection (threat T-27.7-04-06). The `spawnFn` parameter is dependency-injection for test mocking — production code path uses `child_process.spawnSync` unchanged.

### `scripts/lib/install/mcp-register.d.cts` (new)

TypeScript ambient declarations. Sibling kept in sync with the `.cjs` per Phase 27.6 precautionary lesson. Declares `RegisterMcpResult`, `DetectResult`, `HarnessId`, `RegisterMode`, etc.

### `scripts/install.cjs` (modified)

- `helpText()` gains two lines in the Modifiers section: `--register-mcp` (opt-in) + `--no-register-mcp` (default behavior, included for symmetry).
- Main install flow gains a post-install MCP-registration hook that fires only when:
  - not uninstalling
  - not dry-run
  - `--register-mcp` is explicitly passed (D-07 opt-in)
- For each of `['claude', 'codex']`, calls `registerMcp({harness})` and prints one status line per harness:
  - `[install] <harness> CLI not on PATH — skipping gdd-mcp registration` (stderr) when CLI absent
  - `[install] gdd-mcp already registered with <harness> — skipping.` (stdout) for idempotent re-run
  - `[install] gdd-mcp registered with <harness>.` (stdout) for fresh add
  - `[install] gdd-mcp registration with <harness> failed: exit N` (stderr) on failure
- Outer try/catch handles the case where the lib is not yet shipped (forward-compat — silent stderr notice, no install failure).

### `skills/health/SKILL.md` (modified)

Inserts a `<step name="check-mcp-registration">` block before the existing `## Update notice (safe-window surface)` section. The block declares:

1. **Dismissal check** — read `.design/config.json` inside a try/catch; if `mcp_nudge === false`, skip silently. On parse failure: default to `mcp_nudge=true` (show row) — fail-safe per T-27.7-04-05.
2. **Detection** — read `.claude/settings.local.json` directly OR (preferred) invoke `detectMcpRegistration()` from the install-lib.
3. **Row rendering** — one of 5 verbatim strings:
   - `MCP server: registered with claude+codex`
   - `MCP server: registered with claude`
   - `MCP server: registered with codex`
   - `MCP server: not registered  (run: npx @hegemonart/get-design-done --register-mcp; dismiss: .design/config.json#mcp_nudge=false)`
   - `MCP server: unknown (claude/codex CLI not found)`
4. **Fallback** — if `mcp-register.cjs` not yet shipped, render `MCP server: unknown`. Step is explicitly non-blocking.

## Decisions Made

- **D-07 honored**: `--register-mcp` is opt-in (default OFF); `--no-register-mcp` is a no-op today but reserved for symmetry / when the default flips. Auth-gate-style messaging if user runs install without the flag — no surprise mutations to harness config.
- **D-08 honored**: gdd-health MCP-row is one line, dismissable via `.design/config.json#mcp_nudge=false`. Step's dismissal check is the first thing it does — short-circuits before any detection work.
- **Blocker #3 + Warning #9 resolved**: The surface is `skills/health/SKILL.md` (the actual `gdd-health` slash-skill). Production-code grep for `scripts/cli/gsd-health` returns 0 hits in `skills/` + `scripts/`. The 4 hits in `tests/gdd-health-mcp-row.test.cjs` are the guard-test needle itself (the test that *enforces* the absence) — that's correct.
- **spawnFn injection over spawnSync monkey-patching**: keeps the production lib code path identical, no test-side global mutation. The `spawnFn` parameter default value (`spawnSync` from `node:child_process`) means production callers don't need to know about the injection point.
- **HARNESSES is `Object.freeze`d + readonly arrays**: defense in depth for T-27.7-04-06 (flag injection / arg tampering). The plan's `--` separator before `MCP_NAME` is the primary mitigation; freeze is the belt-and-suspenders.
- **Step inserted BEFORE 'Update notice' (not after)**: places the MCP row adjacent to the health table where it conceptually belongs, before unrelated banner content.
