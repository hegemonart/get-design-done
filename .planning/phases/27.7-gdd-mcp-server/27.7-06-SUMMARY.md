## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Replaced plan's greedy cross-line MD038 regex with a proper per-line span scan**

- **Found during:** Task 1 acceptance verification (`node -e ... t.match(/\`\s[^\`]*\`|\`[^\`]*\s\`/g)` exited 1 against a README that has no actual MD038 violations).
- **Issue:** The plan's MD038 detection regex `` `\s[^`]*` `` / `` `[^`]*\s` `` is greedy and matches non-adjacent backtick pairs across lines (e.g., a closing `` ` `` of one inline span followed by surrounding whitespace and content and then an opening `` ` `` of another span, several lines later). That produced 50+ false positives against a README whose actual inline spans are all clean.
- **Fix:** In `tests/gdd-mcp-readme-bench.test.cjs` Test 5, replaced the regex with a per-line scan that (a) skips fenced code blocks (markdownlint MD038 also skips them), (b) uses `/(?<!`)`([^`\n]+)`(?!`)/g` to find true single-line inline spans, and (c) reports a violation only when the span's content has a leading or trailing space. Documented the deviation in the test file's header block and in this SUMMARY.
- **Verification:** Per-line scan returns 0 violations against the committed README. Same scan would correctly flag a string like `` ` foo ` `` if it were introduced.
- **Files modified:** Test file only — the README itself passes both the broken and the corrected check (it has no leading/trailing-space inline spans).
