# Phase 27.7 Plan 05: Skill adoption — progress/resume/next gain MCP-path + File-read-path blocks Summary

Wired the first 3 skill-side callers of the `gdd-mcp` server's 12 tools. Each of `progress`, `resume`, and `next` SKILL.md now forks between an MCP-preferred path (1–3 tool calls, ~3–5s) and a file-read fallback (5–10 reads, ~60–100s), with frontmatter `tools:` widened to advertise the relevant `mcp__gdd_*` entries.

## What shipped

### Task 1 — skills/progress/SKILL.md (commit `bfbe5a1`)
- Frontmatter `tools:` gains `mcp__gdd_status, mcp__gdd_phase_current` (additive — Phase 20 `mcp__gdd_state__get` preserved for the file-read fallback).
- Step 1 (Read state) restructured as a fork:
  - **MCP path (preferred)** — `mcp__gdd_status` returns `{phase, branch, last_decisions, last_completed_plans, blocker_count}` in one call; optional `mcp__gdd_phase_current` adds `stage` + `task_progress`. Total: 1–2 calls, ~3s, ~32k tokens.
  - **File-read path (fallback)** — preserves the legacy `mcp__gdd_state__get` + `Read .design/TODO.md` flow. Total: 5–10 reads, ~100s, ~46.5k tokens.
- Steps 2 (Default output), 3 (Forensic audit), and 4 (Update notice) untouched verbatim.

### Task 2 — skills/resume/SKILL.md + skills/next/SKILL.md (commit `4e371bc`)

**resume/SKILL.md:**
- Frontmatter `tools:` gains `mcp__gdd_status, mcp__gdd_phase_current, mcp__gdd_plans_list, mcp__gdd_decisions_list` (additive — Phase 20 mutator entries preserved).
- New `## Step 0 — Prime cycle context` inserted BEFORE the existing 10 numbered steps. Citing tools:
  - `mcp__gdd_status` (cycle + branch + last decisions)
  - `mcp__gdd_cycle_recap` (diff vs last snapshot — session-restoration signal)
  - `mcp__gdd_decisions_list` (full D-XX list)
  - Optional `mcp__gdd_plans_list` (next incomplete plan)
- File-read fallback reads STATE.md + CYCLES.md.
- Existing Steps 1–10 (parse argument, list checkpoints, restore status, route by stage) untouched.

**next/SKILL.md:**
- Frontmatter `tools:` gains `mcp__gdd_status, mcp__gdd_phase_current, mcp__gdd_plans_list` (was just `Read, Write`).
- `## Logic` section refactored as the fork. MCP path cites `gdd_status` + `gdd_phase_current` + optional `gdd_plans_list` (2–3 calls = full routing decision).
- File-read fallback preserves the `.design/STATE.md` existence check + frontmatter parse.
- Routing table extracted into its own `## Routing table` section (the original numbered "3. Print" became orphan after the fork; promoted to its own `## Output` section).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Missing Critical] next/SKILL.md orphan numbering after fork**

- **Found during:** Task 2 (after editing `## Logic`)
- **Issue:** The original `## Logic` section ended with "3. Print the recommendation as a single formatted block:" — after the fork inserted new sub-headers, the leading "3." became orphaned outside any list context, producing a markdownlint MD029 (ordered-list-item-prefix) violation and reader confusion.
- **Fix:** Promoted "3. Print the recommendation..." to its own `## Output` heading. Cleaner separation; semantically identical.
- **Files modified:** skills/next/SKILL.md
- **Commit:** `4e371bc`

- **STATE.md was modified outside this plan's scope** (presumably by an external session-start hook): only the three SKILL.md files plus the new test were staged and committed. STATE.md changes were NOT committed.

A genuine RED phase is impossible — the headers must exist before the assertion can be valid. Committed under the `test(...)` type per convention; this is documented in `decisions.test-not-true-RED`.
