### Architectural Mapping (not deviations — documented design choices)

**ParsedState shape mapping**: STATE.md (`scripts/lib/gdd-state/types.ts`) does **not** have a top-level `plans` field. The pipeline tracks plan acceptance criteria via the `<must_haves>` block (M-NN entries with status `pending|pass|fail`). Per the plan's own behavior section, `gdd_status.last_completed_plans` and `gdd_plans_list.plans` surface must_haves as the closest analog. The tool returns `{id, name, status}` derived from `{id: m.id, name: m.text, status: m.status}` so consumers see a stable plan-like shape regardless of the underlying STATE schema.

**Phase identifier**: ParsedState's `position` has `stage`, `wave`, `task_progress`, `status` — but no `phase` field. `state.frontmatter.cycle` is the canonical phase identifier in the STATE.md schema (e.g. `cycle: cycle-test`); both `gdd_status.phase` and `gdd_phase_current.phase` read from there.
