## Decisions Made

1. **Surgical Node.js filter over manual edit / sed regex.** `registry.json` is 984 lines with 151 entries. A line-anchored sed pattern would be fragile (entry blocks vary 5–7 lines depending on whether the optional 6th field is present); a JSON parse-filter-serialize round-trip is provably correct against a names whitelist. Re-serializing via `JSON.stringify(data, null, 2) + '\n'` gives identical formatting to the source (verified pre/post: 2-space indent, trailing LF).
2. **Contract §D-06 rewritten as three H3 subsections rather than three bullets.** The plan's intended framing covers three independent concerns (placement classes / migration policy / 28.6 retrospective). H3 subsections give each concern a discoverable anchor (e.g., `#reference-placement-classes`) which downstream skills can deep-link into. Three nested bullets would have lost that addressability.
3. **Kept the original "## Progressive disclosure" H2 heading.** Refreshing the framing under the existing heading preserves the §D-06 anchor identity that other plans and CONTEXT.md entries link to. A renamed heading would have orphaned those cross-links.
4. **Updated the concrete example to mention both placement options.** The original example only mentioned `reference/<framework>-matrices.md`. The refreshed example shows both `reference/` (for 3+ consumers) and `skills/<owner>/` (for 1 consumer), so the example aligns with the new placement classes rather than reinforcing the deprecated centralize-everything stance.
